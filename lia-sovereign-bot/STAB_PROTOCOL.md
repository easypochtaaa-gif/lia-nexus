# Протокол STAB (StabaX) — Спецификация

> Внутренняя экономика бота **LIA Sovereign Core** (Империя STAB).
> Документ описывает, что такое STAB, где он начисляется, как хранится и как
> отображается. Версия отражает текущую реализацию кода (`lia-sovereign-bot`).

---

## 1. Что такое STAB

**STAB** (отображаемое имя — **StabaX**, символ 🪙) — внутриигровая «мягкая»
валюта геймификации Империи STAB. Это **не on-chain токен**: баланс хранится в
базе данных бота как целое число и начисляется за активность пользователя.

Ключевые свойства:

| Свойство | Значение |
|---|---|
| Тип | внутренняя валидируемая валюта (off-chain) |
| Хранение | поле `User.stab_coins` (`Integer`, `default=0`) |
| Делимость | нет, только целые значения |
| Эмиссия | не ограничена, начисляется событиями (квесты, рефералы, действия) |
| Списание | в текущей версии не реализовано (только начисление и просмотр) |
| Связанная метрика | `User.xp` (опыт RPG) начисляется параллельно |

STAB работает в связке с системой **XP** (опыт) и RPG-механиками: квесты,
реферальные битвы, территории и карта наград.

## 2. Модель данных

Все сущности описаны в `database/models.py`.

### 2.1. `User`
- `stab_coins: int` — текущий баланс STAB (баланс пользователя).
- `xp: int` — опыт, начисляется вместе с STAB.
- `referrals_count`, `bonus_requests` — сопутствующие награды реферальной системы.
- `wallet_address` — привязанный Web3-кошелёк (TRC-20 / ETH / Solana).

### 2.2. `Quest`
Шаблон квеста с фиксированной наградой:
- `reward_stab: int` — сколько STAB выдаётся за выполнение.
- `reward_xp: int` — сколько XP выдаётся.
- `is_active: bool` — активен ли квест.

### 2.3. `UserQuest`
Связка «пользователь ↔ квест» со статусом (`active` / `completed`) и временем
выполнения. Используется как защита от повторного получения награды.

### 2.4. `QuestApplication`
Заявка игрока на собственный квест (в т.ч. квест-битва):
- `requested_reward_stab` — запрашиваемая награда в STAB.
- `is_battle`, `opponent_id`, `territory_id`, `ref_invite_new` — параметры
  PvP-механики и захвата территорий.
- `status`: `pending → approved → active → completed` (или `rejected`).

### 2.5. `Territory` и `MapReward`
- `Territory` — захваченная игроком зона на карте (гео-точка + радиус).
- `MapReward` — точка награды на карте (`type`: `coin` / `xp`, поле `amount`).
  Модель подготовлена для механики сбора наград с карты.

## 3. Источники начисления STAB (эмиссия)

Ниже — все места в коде, где баланс STAB увеличивается.

### 3.1. Реферальная система — `handlers/start.py`
При регистрации нового пользователя по реферальной ссылке пригласившему
начисляется бонус:

| Тип приглашения | STAB | Бонусные запросы |
|---|---|---|
| Обычное (`ref_<id>`) | **+500** | +5 |
| Квест-битва (`challenge_<id>_<territory>`) | **+1000** (приз x2) | +10 |

Реферер получает уведомление о начислении. См. `cmd_start` (строки ~66–94).

### 3.2. Привязка Web3-кошелька — `handlers/webapp_handler.py`
За **первую** привязку кошелька (через `/wallet <адрес>` или Mini App):

- **+500 StabaX** и **+100 XP** (только при первой привязке;
  повторная привязка бонус не даёт). См. `cmd_wallet` (строки ~44–52).

### 3.3. Завершение квестов на сайте — `handlers/quests.py`
Команда `/claim <кодовое_слово>`:

- Кодовое слово `LIA_QUEST_COMPLETED_777` → **+1000 StabaX** и **+1000 XP**.
- Защита от повторного получения через запись `UserQuest` (quest_id `777`).
- Связанная команда `/hapanut` выдаёт промежуточный код квеста на сайте
  (`LIA_WANTS_TO_SMOKE_1337`).

### 3.4. Базовые квесты (сидирование) — `main.py`
При первом запуске в БД засеиваются стартовые квесты (`seed_quests`):

| ID | Название | reward_xp | reward_stab |
|---|---|---|---|
| 1 | Подключить Кошелек | 100 | 500 |
| 2 | Первый Контакт | 250 | 100 |
| 3 | Пригласить Рекрута | 500 | 1000 |

> Примечание: шаблоны наград заданы, выдача по ним привязана к механикам
> кошелька/рефералов/`/claim`.

## 4. Просмотр баланса

### 4.1. В боте — `/referral`
Хэндлер `handlers/referral.py` показывает текущий баланс:
`🪙 StabaX: <stab_coins>` вместе с реферальной статистикой и ссылкой.

### 4.2. Через HTTP API (n8n) — `api_server.py`
`GET /api/user/{user_id}` (требует авторизации) возвращает профиль, включая
поле `stab_coins` и `xp`. Используется для дашбордов и интеграций n8n.

---

## 5. Жизненный цикл STAB

```
[Событие активности]
   ├─ Реферал (start.py)        → +500 / +1000
   ├─ Привязка кошелька          → +500 (один раз)
   ├─ /claim код квеста          → +1000
   └─ (план) MapReward / квесты  → amount
            │
            ▼
   User.stab_coins += N   (commit в БД)
            │
            ▼
   Просмотр: /referral  •  GET /api/user/{id}
```

---

## 6. Защита и анти-фрод

- Повторное получение награды за квест блокируется через `UserQuest`
  (`/claim`, проверка `quest_id`).
- Бонус за кошелёк выдаётся только при **первой** привязке (`was_connected`).
- Общие меры: `User.is_banned`, `ip_hash`, лог угроз `Threat` и модуль AEGIS.

---

## 7. Зарезервировано / план развития

Реализовано в модели, но пока не задействовано в хэндлерах — кандидаты на
«пробуждение» (continuation roadmap):

- **Списание STAB**: магазин/апгрейды/премиум за внутреннюю валюту
  (сейчас есть только начисление).
- **MapReward**: сбор монет/XP с гео-точек на карте Mini App.
- **Territory**: захват и удержание территорий, награды за владение.
- **Квест-битвы (PvP)**: полный цикл `QuestApplication.is_battle` с призом x2.
- **On-chain мост**: конвертация StabaX ↔ ончейн-токен через `wallet_address`.
